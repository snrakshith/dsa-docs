Authentication Flow in the Web App#

We’ve implemented OpenID Connect (OIDC) using oidc-client-ts to manage user authentication. Below is an overview of the key concepts and code setup.

1. What is OIDC and why are we using it?#

OIDC (OpenID Connect) is a protocol built on OAuth 2.0, providing identity verification and authentication. We’re using this for:

  • User Authentication: Verify the user’s identity and retrieve their profile and email.
  • Token Management: Secure communication with an authorization server using access tokens.
  • Session Management: Maintain user sessions across browser tabs.

3. How the Configuration Works#

Authentication Settings:#

  1. response_type:

    • We’re using code flow for security. This means we exchange an authorization code for access tokens, preventing the exposure of tokens in the URL.

  2. scope:

    • We’re requesting the user’s profile, email, and openid (basic identity). This helps us retrieve user-specific data like their email or name.

  3. authority:

    • The URL of the identity provider/authorization server (e.g., Auth0, Okta). This will validate the user’s identity.

  4. client_id & client_secret:

    • Identifiers for our web app, registered with the identity provider. These ensure our app is authorized to request user information.

  5. redirect_uri:

    • Where the user is redirected after a successful login. We’re using /dashboard as the landing page.

  6. post_logout_redirect_uri:

    • After the user logs out, we redirect them to the home page (window.location.origin).

Session Management:#

  • userStore:

    • Stores the user’s session in sessionStorage (scoped to the browser tab). This means the session ends when the browser tab closes.

  • monitorSession:

    • Tracks the login/logout events across tabs, so when the user logs out in one tab, they will also be logged out in other tabs.

4. Handling the Sign-In Callback#

After the user signs in, OIDC redirects them back to the redirect URI (in our case, /dashboard). The callback function ensures the page state remains clean.

  • Why is this needed?
    When OIDC redirects the user back, the URL may contain some extra query parameters (?code=...&state=...).
    • The onSigninCallback clears these parameters without reloading the page by using window.history.replaceState.

5. Flow in Action#

  1. Login Flow:

    • User clicks login in the app.
    • They are redirected to the authorization server’s login page.
    • After successful login, they are redirected back to /dashboard.
    • Session is saved in sessionStorage and shared across open tabs.

  2. Logout Flow:

    • When the user logs out, the post_logout_redirect_uri triggers, redirecting them to the home page.
    • If multiple tabs are open, monitorSession ensures all tabs reflect the logged-out state.

6. Key Points to Explain During KT:#

  • Security: Explain the importance of using authorization code flow and how it secures the token exchange process.
  • Session Management: Highlight how WebStorageStateStore and monitorSession help in multi-tab login/logout handling.
  • Environment Variables: Mention how import.meta.env variables are used to keep sensitive information (like client ID) out of the codebase.
  • Callback Management: Emphasize the need for onSigninCallback to clean up the URL after authentication.

7. Possible Issues and How to Debug#

  • Login not working?

    • Check if the client_id and authority URLs are correctly set.

  • Session not shared across tabs?

    • Make sure monitorSession is enabled, and sessionStorage is available in the browser.

  • Redirect loop after login?

    • Ensure the redirect_uri and callback logic are correctly configured.
